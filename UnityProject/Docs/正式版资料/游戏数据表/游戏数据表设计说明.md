# WordSearch 英语 — 游戏数据表设计说明

> 本文档说明所有配表的结构、表间关系、数据读取逻辑、玩家进度方案和活动合成流程。

---

## 一、数据架构总览

整个游戏数据分为三层：

```
本地配置（8张CSV）    → 随版本发布，客户端/CDN分发
服务器配置（15张CSV）  → 服务端动态下发，运营可热更新
关卡JSON文件          → 独立产出物，由关卡设计工具生成
```

关卡JSON包含网格数据、目标词、wordDetails（释义/记忆曲线）、hiddenWords，是完整的关卡运行数据。配表不重复存储关卡JSON中已有的信息。

---

## 二、CSV格式规范

- 编码：UTF-8 with BOM
- 第1行：英文表头（字段名）
- 第2行：中文说明（字段含义）
- 第3行起：数据行
- 空值留空，不填null

## 三、本地配置表（8张）

### 3.1 stage_config — 学段配置表

定义游戏的四个学段，决定玩家的起点和解锁路径。

| 字段 | 类型 | 说明 |
|------|------|------|
| stage_id | string | 学段ID，主键 |
| stage_name | string | 学段中文名 |
| sort_order | int | 排序序号 |
| start_pack_id | string | 起始关卡包ID → pack_config.pack_id |
| unlock_next_stage | string | 完成后解锁的下一学段 → 自关联stage_id |


### 3.2 pack_config — 关卡包配置表

定义每个主题关卡包的元数据。关卡包通过 next_pack_id 形成链表，串联整个学段的关卡流程。

| 字段 | 类型 | 说明 |
|------|------|------|
| pack_id | string | 关卡包ID，主键 |
| stage_id | string | 所属学段 → stage_config.stage_id |
| theme_en | string | 主题英文名 |
| theme_zh | string | 主题中文名 |
| total_levels | int | 总关卡数 |
| word_count | int | 包含单词数 |
| sort_order | int | 在学段内的排序（用于计算display_level） |
| unlock_type | string | 解锁方式：stage_select（选学段即解锁）/ complete_previous（完成前置包解锁） |
| required_pack_id | string | 前置关卡包ID → 自关联pack_id |
| next_pack_id | string | 下一个关卡包ID → 自关联pack_id |

### 3.3 word_pool — 单词总池表

## 六、玩家进度方案

### 6.1 存储方案

玩家进度使用**绝对定位**存储，不存全局关卡号：

```json
{
  "user_id": "xxx",
  "stage": "primary",
  "current_pack_id": "primary_family",
  "current_level_in_pack": 15
}
```

**为什么不存 display_level（全局关卡号）？**

display_level 依赖所有关卡包的顺序和数量。如果某个关卡包增减关卡数、或中间插入/删除关卡包，display_level 的映射就会偏移，导致老玩家进度错乱。

存 pack_id + level_in_pack 是绝对定位，无论其他关卡包怎么变化，玩家的进度永远指向正确的关卡。

### 6.2 display_level 计算（客户端运行时）

display_level 仅用于UI显示（如主界面大按钮"Play Level 35"），由客户端实时计算：

```
1. 加载 pack_config，筛选玩家当前学段及之前已完成学段的所有包
2. 按 stage排序 + sort_order 排序
3. 逐个累加 total_levels，直到 current_pack_id
4. 加上 current_level_in_pack
5. 得到 display_level

示例（小学生，当前 primary_family 第15关）：
  primary_self_intro: sort=1, total=20 → 累计 20
  primary_family: sort=2 → 这是当前包
  display_level = 20 + 15 = 35
```

### 6.3 通关推进逻辑

```
玩家通关当前关卡：
  current_level_in_pack += 1

  if current_level_in_pack > pack的total_levels:
      查 pack_config.next_pack_id
      if next_pack_id 不为空:
          current_pack_id = next_pack_id
          current_level_in_pack = 1
          if next_pack_id 的 stage_id != 当前 stage:
              stage = next_pack_id 的 stage_id  // 学段切换
      else:
          全部通关完毕
```

### 6.4 学段切换示例

```
小学生打完小学最后一个包 primary_numbers 的第20关：
  → next_pack_id = "junior_daily_life"
  → 查 pack_config → stage_id = "junior_high"
  → 更新：stage = "junior_high"
  → current_pack_id = "junior_daily_life"
  → current_level_in_pack = 1
  → display_level 继续累加（小学总关卡数 + 1），不重置
```

## 七、关卡JSON文件组织

### 7.1 目录结构

```
levels/
├── primary_self_intro/
│   ├── primary_self_intro_1.json
│   ├── primary_self_intro_2.json
│   └── ...primary_self_intro_20.json
├── primary_family/
│   ├── primary_family_1.json
│   └── ...primary_family_20.json
└── junior_daily_life/
    └── ...
```

### 7.2 文件定位规则

```
目录 = {pack_id}
文件名 = {pack_id}_{level_in_pack}.json

客户端加载路径：
  levels/{current_pack_id}/{current_pack_id}_{current_level_in_pack}.json
```

### 7.3 关卡JSON与配表的关系

关卡JSON是完整的关卡运行数据，包含：
- 网格数据（gridString、wordPositions）
- 目标词列表（words）
- 学习数据（wordDetails：释义、音标、例句、记忆曲线标记）
- 隐藏词（hiddenWords：含坐标和奖励）
- 难度参数（dimension、useHardDirections等）

配表不重复存储这些信息。配表的作用是：
- **pack_config**：告诉客户端"有哪些关卡包、每个包多少关、怎么串联"
- **word_pool**：提供单词字典查询（学习模式下的额外信息来源）
- **landmark_card_config**：告诉客户端"这个包的第几关解锁哪张景区卡片"

## 八、活动合成流程

活动标记不存在关卡JSON中，是客户端运行时叠加的"透明贴纸"。

### 8.1 完整流程

```
玩家点击"Play Level 35"
  │
  ├── 1. 读取玩家进度
  │   → pack_id = primary_family, level = 15
  │
  ├── 2. 加载关卡JSON
  │   → levels/primary_family/primary_family_15.json
  │   → 目标词: ["FATHER","MOTHER","SISTER","BROTHER","BABY"]
  │
  ├── 3. 请求服务端活动配置（event_config）
  │   → 活动A: 周中收集赛 (need_word_mark=1, mark_ratio=0.4, icon=star)
  │   → 活动B: 头像活动 (need_word_mark=1, mark_ratio=0.2, icon=compass)
  │   → 活动C: 工作日学习赛 (need_word_mark=0) → 不需要标记，跳过
  │
  ├── 4. 客户端合成标记
  │   → 5个目标词
  │   → 活动A: ceil(5 × 0.4) = 2个
  │     用 hash(event_id + pack_id + level_id) 做随机种子选词
  │     → MOTHER⭐, BABY⭐
  │   → 活动B: ceil(5 × 0.2) = 1个，从剩余词中选
  │     → SISTER🧭
  │   → 一个词只能被一个活动标记，按活动优先级依次分配
  │
  ├── 5. 渲染界面
  │   → FATHER, MOTHER⭐, SISTER🧭, BROTHER, BABY⭐
  │
  └── 6. 通关结算
      → +5 学习积分（5个目标词 × 1分）
      → MOTHER⭐ → +1 Frame Point（周中收集赛）
      → BABY⭐ → +1 Frame Point
      → SISTER🧭 → +1 Avatar Token（头像活动）
      → Daily Dash 进度 +1/5
      → Tournament 排名 +学习星
      → current_level_in_pack = 16
```

### 8.2 random_seed 保证公平性

```
seed = hash(event_id + pack_id + level_in_pack)
```

- 同一活动期 + 同一关卡 = 所有玩家看到相同的标记词
- 不同活动期 + 同一关卡 = 标记词会变（因为 event_id 变了）

### 8.3 需要标记目标词的活动

| 活动 | 需要标记 | 标记图标 |
|------|---------|---------|
| 周中收集赛（River Race） | 是 | star（星标） |
| 主题头像活动 | 是 | compass（指南针） |
| 工作日学习赛 | 否 | — |
| 周末冲分赛 | 否 | — |
| 每日冲刺 | 否 | — |
| 每日挑战 | 否（独立关卡） | — |
| 单词大师 | 否 | — |
| 每日登录奖励 | 否 | — |

实际只有2类活动需要标记词，冲突概率很低。
